## Описание проекта

Веб-приложение для отображения информации обо всех сотрудниках компании в виде списка с возможностью фильтрации по департаментам, поиска, сортировки и просмотра детальной информации. Данные загружаются с предоставленного API.

2. Установка зависимостей(React Router, Axios, Redux Toolkit, styled-components);
3. Настройка базовой структуры проекта.

На основе UI создаются компоненты с логикой:

1. Подключаем стор и сервисы.
2. Компоненты берут данные из стора и диспатчат экшены.

Примеры:

2. UserList — отображает список пользователей;

## Виды пагинации

### Классическая пагинация (нумерованные страницы)

1. Как работает:
Запрос к серверу передаёт номер страницы или смещение и количество элементов на странице. 
Пример: ```GET /users?offset=20&limit=10``` -> вернуть 10 пользователей, начиная с 21-го.

2. Плюсы:
  1. Можно сохранить ссылку на страницу;
  2. Хорошо работает в таблицах и каталогах;
  3. Просто в реализации.

3. Минусы:
  1. Не всегда удобно на мобильных устройствах — кнопки могут быть мелкими;
  2. При частых вставках/удалениях записи "плавают" между страницами;
  3. На больших данных может тормозить, а при быстрой прокрутке казаться медленной.

### "Load more" (кнопка "Показать еще")

1. Как работает:
Пользователь нажимает кнопку "Показать ещё", сервер подгружает следующую порцию данных.

2. Плюсы:
  1. Нет перезагрузки страницы;
  2. Простая для пользователя, не нужно кликать на номера страниц;
  3. Легче UX, чем у классической.

3. Минусы:
  1. Пользователь не видит полный размер контента;
  2. Хуже для быстрого перехода в середину/конец списка;
  3. Сложнее вернуться к конкретной позиции;
  4. Нельзя "сохранить" позицию — если перезагрузить страницу, придётся листать заново.

### Бесконечная прокрутка (Infinite Scroll)

1. Как работает:
Элементы подгружаются по мере прокрутки пользователем.

2. Плюсы:
  1. Отличный UX для лент контента (например, соцсети, новости);
  2. Минимальные действия со стороны пользователя.

3. Минусы:
  1. Пользователь не видит "конец" списка, сложно ориентироваться;
  2. Сложнее реализовать навигацию "назад/вперед";
  3. Сложнее реализовать — нужна отслеживаемая зона прокрутки и логика подгрузки.

### Курсорная пагинация (Cursor-based Pagination)

1. Как работает:
Сервер возвращает элемент, с которого начинать следующую страницу (курсора).
Пример: запрос вида ```/posts?cursor=abc123&limit=10```, где cursor — ID последнего элемента.

2. Плюсы:
  1. Очень эффективна при больших данных — не нужно "пропускать" записи, используется индекс;
  2. Быстрая навигация даже при большом объёме информации;
  3. Идеально для лент, чатов, бесконечной прокрутки.

3. Минусы:
  1. Нельзя просто так "перейти на страницу 7" — навигация линейная, шаг за шагомv;
  2. Требует хранения курсора (например, ID последнего элемента);
  3. Сложнее реализовать — нужна отслеживаемая зона прокрутки и логика подгрузки.

### Выводы

Были перечислены основные способы реализации пагинации, конечно, каждый из них хорошо работает при определенных условиях:

1. Нумерованные страницы -> если нужна четкая структура и навигация (каталоги, таблицы);
2. "Load more" -> для соцсетей, новостных лент, где контент "потребляется" пачками;
3. Бесконечная прокрутка -> так же, как и в "Load more", но важна непрерывность контента, чтобы от пользователя не требовалось доп. действий(нажать на кнопку);
4. Курсорная пагинация -> если данных много, и они часто обновляются, например, чат
